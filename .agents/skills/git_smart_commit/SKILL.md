# Git Smart Commit Skill

このスキルは、未コミットの変更（ステージング済み・未ステージング）を分析し、意味のある論理的な単位（粒度）に分割してコミットを作成し、最終的にリモートリポジトリへ push するためのものです。

## 実行手順

このスキルがトリガーされた場合、以下の手順に従って実行してください。

### 1. 変更内容の確認
1. `git status` を実行して、変更されたファイル、新規ファイル、削除されたファイルを把握します。
2. `git diff`（およびステージング済みの差分を確認するために `git diff --cached`）を実行し、具体的な変更内容（コードの差分）を確認します。

### 2. コミット単位の設計 (コミットプランの作成)
変更内容を論理的な単位に分割します。例えば以下のように分類します：
- `feat`: 新機能の追加
- `fix`: バグの修正
- `refactor`: 仕様変更を伴わないコードの整理
- `docs`: ドキュメントの更新
- `style`: コードのフォーマット修正（動作に影響しないもの）
- `test`: テストコードの追加・修正
- `chore`: ビルドプロセスやライブラリのアップデート、設定ファイルの変更など

1つのファイルに複数の異なる目的の変更が含まれている場合、可能であればファイルを分割して add するか、論理的に関連する変更ごとにコミットを分けます。
**コミットプラン**をマークダウンのリスト形式で作成し、ユーザーに変更内容とコミットメッセージ案を提示して確認を求めます。

### 3. コミットの実行
ユーザーからプランの承認を得た後、または自己判断の指示がある場合、各コミットを順次実行します。
1. `git add <file>` または必要な範囲を指定してステージングします。
2. わかりやすく、簡潔で具体的なコミットメッセージを作成して `git commit -m "<message>"` を実行します。
   - メッセージは Conventional Commits 形式（例: `feat(auth): add login validation`）またはプロジェクトの規約に従ってください。

### 4. プルとプッシュ
1. コミットがすべて完了したら、`git pull --rebase` などを実行してリモートの最新の変更を取り込みます。
2. `git push origin <current-branch>` を実行してリモートリポジトリに送信します。

## 注意事項
- 意図しないファイルをコミットに含めないよう、`.gitignore` に従っているか確認してください。
- 競合が発生した場合は、速やかにユーザーに報告し、解決手順を確認してください。
